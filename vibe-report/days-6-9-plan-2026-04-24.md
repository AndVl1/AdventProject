# План задач: Дни 6–10

Стек: Python. Inference — локально через `llama.cpp` (`llama-server`, OpenAI-совместимый `/v1/chat/completions`).

**Доступные модели**:
- **MoE-large** (имеется): `unsloth/Qwen3-30B-A3B-GGUF:UD-Q3_K_S` — MoE 30B / 3B активных. Используем как Tier 2 в дне 8 + как teacher для генерации синтетики дня 6.
- **FT-target** (скачать в mlx-формате): `mlx-community/Qwen2.5-3B-Instruct-bf16`. Тренируется LoRA через mlx-lm на M4 Pro 48 GB за 10–15 мин, после fuse конверт в GGUF Q4_K_M — Tier 1 дня 8.

**Почему не FT 30B MoE напрямую**: для SFT нужна разкванченная база (≥60 GB на диск), MoE-роутер плохо тренируется LoRA на consumer-железе, qlora-стек для Qwen3 MoE сырой. Гибрид «маленькая FT + большая base teacher» даёт измеримый прирост на дне 6 и реальный routing на дне 8.

**FT-путь**: HF-base → LoRA через `mlx-lm` (Apple Silicon) или `unsloth` (CUDA) → fuse → `convert_hf_to_gguf.py` → `llama-quantize Q4_K_M` → `llama-server`.

## День 6. Dataset + клиент fine-tuning

### Выбор задачи
Классификация тикетов поддержки по категориям (`bug | feature_request | billing | how_to | other`) + тональность (`neg | neu | pos`). Причина: легко мерять accuracy, легко собирать реальные данные (открытые трекеры GitHub Issues, форумы), формат строгий — годится для дней 7–9.

Эталонный ответ assistant — JSON: `{"category": "...", "sentiment": "..."}`. Жёсткий формат → база для constraint-check на дне 7.

### Сбор данных (≥50, реальных ≥20%)
- Реальные (≥10): GitHub Issues из 2–3 публичных репо (Ktor, kotlinx.serialization) через `gh api`. Разметка категорий — вручную (быстро для 10–15 шт).
- Синтетика (≤40): через локальную **Qwen3-30B-A3B** (teacher), `temperature=0.8`, промпт «сгенерируй тикет категории X с тональностью Y, тема: <random>». Темы — из списка 20–30 заготовок. assistant-эталон собираем тем же teacher, но с `temperature=0` + GBNF на JSON-схему.

Формат строки JSONL:
```json
{"messages":[
  {"role":"system","content":"Классифицируй тикет. Ответ строго JSON: {\"category\":\"...\",\"sentiment\":\"...\"}"},
  {"role":"user","content":"<текст тикета>"},
  {"role":"assistant","content":"{\"category\":\"bug\",\"sentiment\":\"neg\"}"}
]}
```

### Очистка (`build_dataset.py`)
- dedupe по нормализованному `user.content` (lowercase, trim, collapse whitespace) + по hash.
- фильтр: `len(user.content)` в диапазоне `[40, 1500]` символов; `assistant.content` парсится как JSON и проходит схему.
- сплит train/eval = 80/20, стратифицированно по `category` (sklearn `train_test_split(stratify=...)`).

### Валидатор (`validate.py`)
Для каждой строки проверяет:
1. Валидный JSON.
2. Ключ `messages` — массив длины 3.
3. Роли по порядку: `system`, `user`, `assistant`.
4. `content` непустой и тип `str`.
5. `assistant.content` парсится как JSON и матчит enum категорий/тональности.
Выход — exit code ≠ 0 при первой ошибке + строка/индекс. Запускать на `train.jsonl` и `eval.jsonl`.

### Baseline
- 10 примеров из `eval.jsonl`.
- Прогон через `llama-server -m Qwen2.5-3B-Instruct-Q4_K_M.gguf` (FT-target, ДО тренировки), `temperature=0`, `top_p=1`.
- Параллельно — прогон через Qwen3-30B-A3B (teacher) для контекста «потолок без FT».
- OpenAI Python SDK с `base_url=http://localhost:8080/v1`, `api_key="local"`.
- `results/baseline.jsonl`: `{input, expected, base_3b_answer, teacher_30b_answer, parsed_ok, category_match, sentiment_match, latency_ms, tokens_in, tokens_out}`.
- После FT повторяем тем же скриптом — сравниваем «3B до» vs «3B после FT» vs «30B teacher».

### Критерии «стало лучше»
- **Accuracy по category** (точное совпадение).
- **Accuracy по sentiment**.
- **Format validity** — % ответов, парсящихся в JSON-схему без ремонта.
- **Avg latency ms/request** (локально — единственная «цена», cost = 0).
- **Tokens/sec** на твоём железе — фиксируем для сравнения после FT.
- Качественное: нет лишнего текста вокруг JSON, стабильность enum-значений.

### Клиент fine-tuning (`ft_client.py`, не запускать)

Локальный LoRA-пайплайн. Стек выбираем по железу:
- **Mac (Apple Silicon)**: `mlx-lm` — `pip install mlx-lm`.
- **CUDA**: `unsloth` — быстрее всего на 7B и ниже.

CLI-обёртка:

```
python ft_client.py prepare      # конверт data/train.jsonl + eval.jsonl в формат тренера (chat template)
python ft_client.py train        # запуск LoRA SFT, логи в results/train.log
python ft_client.py status       # читать последние строки лога + train/val loss
python ft_client.py merge        # merge LoRA в base, выгрузка в models/ft-merged/
python ft_client.py to_gguf      # convert_hf_to_gguf.py + llama-quantize -> models/ft.gguf
python ft_client.py serve        # llama-server -m models/ft.gguf --port 8081
```

Конкретные команды внутри (mlx вариант):

```bash
# 1. LoRA train (M4 Pro 48 GB, ~10-15 мин на 600 iters)
mlx_lm.lora \
  --model mlx-community/Qwen2.5-3B-Instruct-bf16 \
  --train --data data/ \
  --iters 600 --batch-size 4 --lora-layers 16 \
  --learning-rate 1e-5 \
  --adapter-path adapters/

# 2. Fuse adapter в base
mlx_lm.fuse \
  --model mlx-community/Qwen2.5-3B-Instruct-bf16 \
  --adapter-path adapters/ \
  --save-path models/ft-merged/

# 3. Конверт в GGUF + квантизация
python llama.cpp/convert_hf_to_gguf.py models/ft-merged/ \
  --outfile models/ft.gguf --outtype f16
./llama.cpp/llama-quantize models/ft.gguf models/ft-q4km.gguf Q4_K_M
```

Формат данных для mlx-lm: `data/train.jsonl` и `data/valid.jsonl` (имя именно `valid`, не `eval`). Каждая строка — `{"messages":[...]}` (chat-format поддерживается mlx-lm с 0.20+).

Polling: `tail -f results/train.log`, парсим `Iter N: train loss X, val loss Y`, пишем в `results/train_metrics.jsonl`. Стоп по `iters` или ранней остановке (val loss растёт 3 проверки подряд).

В день 6 — только код и dry-run на 5 итерациях. Полный train откладывается.

## День 7. Confidence + контроль качества

База — та же задача классификации. Без FT.

### Подходы (берём 3)

**1. Constraint-based** (обязательный):
- `pydantic` модель `Ticket(category: Literal[...], sentiment: Literal[...])`.
- В `llama.cpp` — GBNF grammar через параметр `grammar` в `/v1/chat/completions` (или `json_schema` в новых билдах llama-server).
- FAIL если не парсится / enum невалиден.

**2. Self-check** (chain-of-verification lite):
- Этап А: модель отдаёт ответ + краткое обоснование (`reason`).
- Этап Б: тот же запрос новым вызовом — «вот тикет, вот предложенная классификация, согласен? верни {agree: bool, fix?: ...}».
- Расхождение → статус `UNSURE`.

**3. Redundancy / self-consistency**:
- 3 независимых вызова с `temperature=0.7`.
- Majority vote по `category`.
- Если все 3 совпали → `OK`. 2/3 → `UNSURE`. Все разные → `FAIL`.

### Сводный статус
`status = OK` если constraint pass И (self-check.agree ИЛИ redundancy.unanimous). Иначе `UNSURE`. `FAIL` если constraint провалился после ретрая.

### Тестовые наборы
- Корректные: 10 из eval.
- Пограничные: 5 двусмысленных тикетов («не работает кнопка, но я хочу чтобы было синим» — bug+feature).
- Шумные: 5 — опечатки, эмодзи, смесь языков, очень короткие.

### Метрики (`results/confidence_runs.jsonl`)
- кол-во `OK / UNSURE / FAIL`.
- сколько запросов потребовали ретрая (constraint fail → 1 ретрай с remediation prompt).
- avg latency: monolithic vs (constraint+self-check+redundancy).
- cost: умножаем токены на тариф, считаем delta vs baseline.

### Артефакты
`scripts/confidence.py`, отчёт `results/day7-report.md` со сводной таблицей по 20 примерам.

## День 8. Routing между моделями

### Стратегия
Два процесса `llama-server` (3-тирный каскад появится в день 10 с добавлением micro):
1. **Tier 1**: FT-модель из дня 6 — `models/ft-q4km.gguf` (Qwen2.5-3B FT) на `:8080`. Узко-специализирована на нашей задаче.
2. **Tier 2**: `unsloth/Qwen3-30B-A3B-GGUF:UD-Q3_K_S` на `:8081`. Широкая модель для пограничных случаев.

Примечание: `router.py` пишем расширяемо — список `tiers: list[Tier]` с общим интерфейсом `predict(text) -> (answer, confidence, meta)`. В день 10 просто добавится `Tier 0 = micro` в начало списка.

«Cost» в локальном сетапе = wall-clock latency и VRAM/RAM, а не $. Меряем именно их.

### Эвристики эскалации (комбинируем)
- **Constraint fail** на Tier 1 → сразу Tier 2.
- **Confidence из дня 7**: если `redundancy` на Tier 1 даёт не-unanimous → Tier 2.
- **Длина входа**: `len(user.content) > 800` символов или `>5` предложений → сразу Tier 2 (мелкая модель плохо держит контекст).
- **Logprobs**: `llama-server` поддерживает `logprobs: true` + `top_logprobs: N`. Средний logprob по токенам ответа < порога (например `-1.0`) → Tier 2.

### Тест
- 30 запросов: смесь простых (10), средних (10), сложных/шумных (10) — переиспользуем выборку дня 7.
- Логируем по каждому: `tier_used`, `escalated`, `reason`, `latency_ms`, `tokens_in/out`, `cost_estimate`.

### Метрики
- % остался на Tier 1 (цель ≥60% на простых).
- Accuracy итоговая vs «всё на Tier 2» (просадка ≤2 п.п.).
- Latency saving % (sum router vs sum «всё на Tier 2»).
- p50/p95 latency.
- Token throughput суммарно.

### Артефакты
`scripts/router.py`, `results/router_runs.jsonl`, табличка cost/accuracy/latency.

## День 9. Multi-stage inference

### Задача
Усложняем: парсинг тикета в структуру `{category, sentiment, severity, affected_component, suggested_action}`. Один запрос плохо тянет — модель путает severity и sentiment, выдумывает component.

### Вариант A (monolithic, для сравнения)
Один промпт «верни JSON со всеми 5 полями». Замер accuracy по каждому полю.

### Вариант B (multi-stage)
- **Этап 1 — нормализация** (Tier 1 = FT-3B):
  - Вход: сырой тикет.
  - Выход (TOON-стиль, компактный):
    ```
    lang=ru
    summary=<≤120 chars>
    keywords=k1,k2,k3
    has_error_msg=true|false
    ```
  - Жёсткий формат через GBNF grammar в `llama.cpp`.

- **Этап 2 — классификация** (Tier 1 = FT-3B):
  - Вход: только нормализованный блок этапа 1.
  - Выход enum-only, по одному вызову на поле:
    - `category` ∈ {bug, feature_request, billing, how_to, other}
    - `sentiment` ∈ {neg, neu, pos}
    - `severity` ∈ {low, medium, high, critical}
  - Каждый вызов — `max_tokens=4`, `logit_bias` или GBNF на enum.

- **Этап 3 — извлечение + решение** (Tier 2 = Qwen3-30B-A3B если уверенность низкая, иначе Tier 1):
  - Вход: исходный тикет + результаты этапов 1–2.
  - Выход: `affected_component` (free text, ≤40 chars) + `suggested_action` (enum: `escalate_eng | reply_template | request_more_info | close`).
  - Здесь подключаем routing из дня 8: если этап 2 дал `severity=critical` или `category=billing` → Tier 2 принудительно.

### Замер
По 20 примерам (микс из дней 6–7):
- Accuracy per-field: monolithic vs multi-stage.
- Total latency (sum по этапам vs один запрос).
- Total cost.
- Кол-во вызовов и распределение по моделям.

Ожидание: multi-stage даёт +accuracy на `severity` и `suggested_action` за счёт изоляции решений, total cost ниже за счёт Tier 1 на этапах 1–2, latency может вырасти — фиксируем честно.

### Артефакты
`scripts/multistage.py`, `results/multistage_runs.jsonl`, сравнительная таблица A vs B.

## День 10. Micro-model first

### Задача
Та же — классификация тикетов (`category` + `sentiment`). Большинство тикетов решается без LLM: короткие шаблонные фразы хорошо ловятся embedding'ами + линейным классификатором. LLM остаётся для пограничных случаев.

Встраивается в существующий каскад как **Tier 0** перед FT-3B и 30B MoE. Итоговая цепочка: `micro → FT-3B → Qwen3-30B-A3B`.

### Выбор micro-модели — 3 варианта для сравнения

Реализуем и меряем все три, финально выбираем победителя по accuracy-on-easy × latency.

**Вариант A — sklearn на TF-IDF** (baseline, самый дешёвый):
- `TfidfVectorizer(ngram_range=(1,2), max_features=5000)` + `LogisticRegression(multi_class='multinomial')`.
- Обучаем два отдельных классификатора: один на `category`, второй на `sentiment`.
- Датасет — `train.jsonl` из дня 6 (50 примеров). Маловато для TF-IDF, но на 5 классов category и 3 classes sentiment сойдёт для демонстрации.
- Confidence = `max(predict_proba)`.
- Latency: ~1 ms/запрос на CPU.

**Вариант B — embeddings + классификатор** (основной):
- Embedding-модель: `sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2` (118 MB, многоязычная — тикеты могут быть на русском) либо `BAAI/bge-small-en-v1.5` если тикеты en-only.
- Поверх — `LogisticRegression` либо `k-NN` (k=5) с cosine similarity.
- Confidence для kNN = доля ближайших соседей в majority-классе. Для LogReg = `max(predict_proba)`.
- Latency: ~10–20 ms/запрос на CPU, ~3 ms на MPS.

**Вариант C — tiny LLM через llama.cpp**:
- `Qwen3-0.6B-Instruct-Q4_K_M` (~400 MB) на `:8079`.
- GBNF grammar на enum категорий → гарантированный формат.
- Confidence = avg logprob ответа (через `logprobs: true`).
- Latency: ~50–100 ms на M4 Pro.

### Порог уверенности и fallback-логика

```
pred, conf = micro.predict(text)

if conf >= THRESHOLD_HIGH:            # 0.85 для LogReg, -0.5 для logprob
    return pred, tier="micro"
elif conf >= THRESHOLD_MEDIUM:        # 0.60
    # спорный — проверяем FT-3B (Tier 1)
    return tier1.predict(text)
else:
    # совсем не уверен — сразу на 30B
    return tier2.predict(text)
```

Пороги подбираем по calibration curve на eval: ищем точку, где precision@micro ≥ 95%.

### Адаптация существующего кода

- **day 6 датасет**: уже содержит `train.jsonl` — используем как есть для обучения micro. Дополнительно синтетику день 6 можно расширить (больше данных = лучше micro), но 50 достаточно для proof-of-concept.
- **day 7 confidence**: constraint-check применяется и к micro-ответу (enum-валидация тривиальна для sklearn — она уже в `Literal` типах pydantic).
- **day 8 router**: `router.py` расширяется списком тиров — micro добавляется нулевым элементом. Эвристики эскалации не меняются, только порог теперь двуступенчатый (HIGH / MEDIUM).
- **day 9 multi-stage**: этап 1 «нормализация» теперь может делаться micro'й (классификация интента как предварительный gating), этап 2 — FT-3B с контекстом от micro.

### Тест
30 запросов из расширенной выборки:
- 10 простых (короткие шаблонные: «не работает логин», «хочу тёмную тему»).
- 10 пограничных (микс категорий, ирония).
- 10 сложных (длинные, многоязычные, с кодом).

Для каждого логируем: `micro_pred, micro_conf, tier_chosen, final_answer, latency_per_tier_ms, total_latency_ms, correct`.

### Метрики (`results/micro_runs.jsonl` + `results/day10-report.md`)
- **Coverage** = % запросов, обработанных только micro (без fallback).
- **Precision@micro** = accuracy среди обработанных micro. Цель ≥95%.
- **Fallback rate по тирам**: % → Tier 1, % → Tier 2.
- **Total LLM calls saved** = 1 − (calls_tier1 + calls_tier2) / N. Цель ≥60% экономия.
- **Avg latency** vs «всё на FT-3B» vs «всё на 30B».
- **Таблица сравнения 3 вариантов micro** (A/B/C): accuracy, latency, размер модели.

## Сквозные решения

- **Конфиг**: `.env` с `TIER1_BASE_URL`, `TIER2_BASE_URL`, `TIER1_MODEL`, `TIER2_MODEL`.
- **HTTP-клиент**: `OpenAI(base_url=..., api_key="local")` — `llama-server` совместимый endpoint.
- **Логирование**: JSONL в `results/`: `ts, stage, model, tokens_in, tokens_out, latency_ms, status, escalated`. Сводка через `pandas`.
- **Метрика «cost»**: считаем wall-clock `latency_ms` + `tokens_out / latency_s` (throughput). Долларов нет.
- **Запуск llama.cpp** (две тачки/два процесса, разные порты):
  ```bash
  # Tier 1 — FT 3B
  ./llama-server -m models/ft-q4km.gguf \
    --host 127.0.0.1 --port 8080 --n-gpu-layers 999 --ctx-size 4096

  # Tier 2 — большой MoE (имеется)
  ./llama-server -m models/Qwen3-30B-A3B-UD-Q3_K_S.gguf \
    --host 127.0.0.1 --port 8081 --n-gpu-layers 999 --ctx-size 8192
  ```
  Учти RAM/VRAM: Tier 2 ≈ 14–16 GB. Если железо тянет один процесс — гоняй по очереди, роутер дёргает их через адаптер с lazy-spawn.
  Grammar передаём per-request в теле (`grammar` поле), не флагом — иначе сервер залочен на одну схему.
- **Видео**: `asciinema` на каждый день + просмотр итогового JSONL.
