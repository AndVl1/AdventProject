# Критерии оценки FT (день 6)

## Метрики

| Метрика | Что меряем | Способ |
|---------|-----------|--------|
| `parsed_ok` | % ответов, парсящихся как валидный JSON по схеме `{category, sentiment}` | `json.loads` + проверка ключей |
| `category_acc` | Точное совпадение `category` с эталоном | exact match по строке |
| `sentiment_acc` | Точное совпадение `sentiment` с эталоном | exact match по строке |
| `joint_acc` | Обе метки совпали одновременно | `cat_match AND sent_match` |
| `avg_latency_ms` | Среднее время одного запроса | `time.perf_counter` per-request |
| `avg_tokens_out` | Средний размер выхода | `usage.completion_tokens` |

## Baseline (до FT)

| Метрика | Tier 1 (Qwen2.5-3B Q4_K_M) | Tier 2 (Qwen3.6-35B-A3B UD-Q3_K_S) |
|---------|---------------------------|------------------------------------|
| `parsed_ok` | 10/10 (100%) | 10/10 (100%) |
| `category_acc` | 2/10 (20%) | 3/10 (30%) |
| `sentiment_acc` | 9/10 (90%) | 9/10 (90%) |
| `avg_latency_ms` | 438 | 708 |
| `avg_tokens_out` | 14.4 | 14.2 |

## Критерии «стало лучше» после FT

FT считается успешным если на тех же 10 примерах из `valid.jsonl`:

1. **`parsed_ok` ≥ 100%** — формат не должен сломаться (GBNF гарантирует, плюс модель должна научиться отдавать чистый JSON и без grammar).
2. **`category_acc` ≥ 55%** — рост минимум +35 п.п. с baseline. Низкий порог потому что синтетика зашумлена (см. наблюдения).
3. **`sentiment_acc` ≥ 85%** — допускается лёгкая просадка (катастрофическое забывание), но не больше -5 п.п.
4. **`joint_acc` ≥ 50%** — обе метки одновременно правильны хотя бы в половине случаев.
5. **`avg_latency_ms` ≤ 600** — FT-результат не должен быть медленнее raw 3B существенно (тот же quant Q4_K_M).
6. **Стабильность формата без grammar**: дополнительный прогон с `--no-grammar` должен дать `parsed_ok ≥ 90%` (если parsed_ok < 50% — модель переобучилась только на grammar-constrained вывод).

## Качественные критерии

- Нет лишнего текста вокруг JSON (никаких "Вот результат:", markdown-кодфенсов).
- Нет hallucinated значений вне enum (`{"category": "issue"}` или `{"sentiment": "negative"}`).
- На пограничных случаях (`other`, `feature_request+neg`) FT-модель должна выдавать корректную метку чаще baseline.

## Наблюдения по данным

- Синтетика зашумлена по `category`: teacher (Qwen3.6-35B-A3B) на инференсе не согласен с метками, которые сам же присвоил при генерации (особенно для пар `billing+pos`, `other+pos`, `bug+pos`).
- Категория `other` плохо отделима — оба тира скатываются в `bug` при `neg` тональности.
- Sentiment задача проще category — оба тира уже на 90%, потолок близко.

Эти наблюдения учтены в порогах: category_acc цель умеренная (55%), sentiment_acc — почти baseline (85%).
